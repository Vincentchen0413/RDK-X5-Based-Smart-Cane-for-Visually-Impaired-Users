# 系统融合架构说明

## 1. 为什么不能只把四套代码放在同一目录

四个功能分别运行时可能都能工作，但直接并行启动会出现以下问题：

- 建图与定位脚本可能同时占用相机、IMU、数据库和 TF；
- 目标识别、导航、低电量、查询和跌倒模块可能同时播报；
- 跌倒模块不知道 RTAB-Map 当前的最新位置；
- 语音模块需要了解系统正在建图还是导航；
- 任一进程异常退出后无人恢复；
- 开机时设备、模型和 ROS 节点启动顺序不确定。

因此需要独立的“融合协调层”，而不是把算法文件简单合并成一个大程序。

## 2. 推荐运行链路

### 2.1 建图

1. 相机和 IMU 驱动发布数据。
2. OpenVINS 输出高频视觉惯性里程计。
3. RTAB-Map 使用图像、深度和里程计建立地图。
4. `location_cache_node.py` 持续缓存当前位置。
5. 用户说“这里是电梯”。
6. ASR 输出 `mark_landmark` 意图和 `place=电梯`。
7. `voice_command_router.py` 发布系统请求。
8. `system_state_manager.py` 调用 `landmark_cli.py add 电梯`。
9. 地点名称、坐标、朝向、地图 ID 和时间被保存。

### 2.2 导航

1. 系统载入已有 RTAB-Map 数据库并进入定位模式。
2. 用户说“我要去电梯”。
3. 意图路由器生成 `navigate` 请求。
4. 状态管理器调用实时导航脚本。
5. 规划器读取当前位置、目标地标和可通行栅格。
6. A* 或其他规划器生成路径。
7. 导航节点持续比较当前位姿和路径，产生直行、左转、右转、到达等事件。
8. 安全事件路由器将导航事件送入音频优先级队列。
9. 如果同时出现水洼或跌倒，危险/紧急事件抢占普通导航播报。

### 2.3 目标识别

1. 板端推理节点读取相机图像。
2. BPU 执行量化模型。
3. 后处理获得类别、置信度和检测框。
4. 双目深度或点云估计目标距离。
5. 事件发布到 `/smart_cane/perception/event`。
6. 安全路由器根据类别、距离、置信度和冷却时间决定是否播报。

### 2.4 跌倒报警

1. `fall.py` 读取 IMU 并执行状态机判断。
2. 检测到异常后写入历史记录或直接发布 ROS 事件。
3. 文件适配器把旧格式转换为统一跌倒事件。
4. 状态管理器进入 `EMERGENCY`。
5. 音频管理器立即播报紧急提示。
6. 安全路由器读取统一位置缓存并向云端/推送服务发送报警。
7. 监护人 App 显示设备、时间、位置和地图跳转入口。

## 3. 资源与优先级

- 相机图像只采集一次，建议通过 ROS 话题共享，避免多个进程重复打开设备。
- OpenVINS 和 RTAB-Map 共用相机/IMU时间戳，必须统一时钟。
- 建图和定位需要互斥。
- 所有语音输出必须经过一个音频管理器。
- 跌倒报警应能在目标检测或导航故障时独立工作。
- 网络不可用时，本地识别、定位和语音仍应运行，报警进入重试队列。
